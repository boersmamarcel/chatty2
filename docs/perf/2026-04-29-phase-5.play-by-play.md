# Phase 5 play-by-play

1. With `stream_smoothing_v1` admitted by deterministic 50/50 seed, provider chunks enter a client-side buffer after first token arrival.
2. Actual TTFT state flips before smoothing; smoothing only affects subsequent paints.
3. A GPUI/Tokio 16 ms frame loop drains `clamp(buffer.len()/4, 1, 3)` characters per frame.
4. Buffer exhaustion keeps the current paint; no spinner is introduced.
5. Stream finalization flushes any remaining buffer so total drift is bounded by pending buffered content.
6. Reduced motion bypasses buffering entirely.

Synthetic metrics: no browser `requestAnimationFrame` exists in GPUI; the desktop fallback uses a 16 ms async frame loop. Constants covered in tests: 30 cps baseline, 90 cps accelerated, acceleration threshold >80 chars.
